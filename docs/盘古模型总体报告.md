# 盘古AI模型总体报告

## 概述

**盘古AI** 是一个面向中文网络小说创作的工程化写作辅助系统，它不是一个简单的"写作助手"——而是一套完整的、可命令驱动的 **小说创作流水线**。系统以项目为基本组织单元，提供从灵感搜集、大纲规划、章节写作、质量分析到改写润色的全流程工程化支持。

本报告汇总了系统 v3.0 版本（截至 2026-06-09）的完整架构、核心能力、参考资源、诊断结果与工程特征。

---

## 一、系统架构

### 1.1 主入口与运行方式

系统主入口为 pangu.py，通过命令行参数驱动：

| 命令 | 功能 |
|------|------|
| `status` | 系统健康检查：列出所有知识模块状态与项目清单 |
| `analyze <项目名>` | 对项目各章节进行文本诊断（TDE评分 + 贝叶斯 + COX处方） |
| `write <项目名> <章节>` | 在现有项目中撰写新章节 |
| `rewrite <项目名> <章节>` | 对已有章节进行多轮重写（多Pass流水线模式） |
| `list` | 列出所有项目目录 |
| `search <关键词>` | 搜索 CSV 参考资料库（通过 reference_engine） |

### 1.2 五大核心知识模块

系统在 knowledge/ 目录下实现了 5 个并列运行的分析引擎，每条文本都会同时经过全部引擎的检查，最终输出一份合成报告。

#### 1.2.1 TextDiagnosticEngine（文本诊断引擎）
- **功能**: 对文本进行 **11 维** 结构化特征分析
- **评分维度**: 动作密度、情绪强度、对话占比、句长变化、平均句长、词汇独特性、句式多样性、语言自然度、信息复杂度、信息丰富度、情绪波动、段落长度、钩子强度
- **评分范围**: 0–100 分（>70 优秀，>50 合格，<50 需要改进）
- **额外检查**: 章末钩子检测、AI模板词识别、重复性词汇警示

#### 1.2.2 BayesianAnalyzer（贝叶斯分析器）
- **功能**: 根据历史样本计算文本的 posterior quality score
- **应用**: 对章节进行"是否需重写"的概率判断，作为自动化决策的依据

#### 1.2.3 EmpiricalBayesAnalyzer（经验贝叶斯分析器）
- **功能**: 对贝叶斯结果进行 shrinkage 校正，避免小样本时过度自信

#### 1.2.4 CoxPrescriptionEngine（COX处方引擎）
- **功能**: 把章节质量问题视为"需介入的风险事件"，输出具体的改写建议列表
- **处方类型**: 增强情绪描写、补充悬念钩子、调整句长波动、减少模板化表达、增加动作密度

#### 1.2.5 ReferenceEngine（参考资源引擎）- **v2.0 新模块**
- **功能**: 统一管理所有参考资源，提供关键词检索与模板获取
- **特色**: 零数据库依赖、纯文件系统、即开即用

### 1.3 后端架构

系统提供了两套后端实现：

- **Python后端**: FastAPI + RAG 知识库 + 风格库 + 文本梯度优化 + 可观测性
- **Java后端**: Spring Boot，通过 PythonBridge（subprocess 调用 pangu.py）提供项目管理与文本分析 API

这种"Java管接口，Python算内容"的混合架构，既保证了企业级部署的稳定性，又保留了自然语言处理生态的灵活性。

## 二、写作工作流（Workshops）

系统将小说创作拆解为 5 个标准化 **Workshop**，每个 Workshop 对应一个独立的 system prompt 与改写策略：

| Workshop | 阶段 | 核心任务 |
|----------|------|----------|
| Workshop 0 | 锚定基调 | 确定故事的第一句、核心冲突与叙事视角 |
| Workshop 1 | 世界观与设定 | 构建世界规则、力量体系、角色群像 |
| Workshop 2 | 章节初稿 | 快速生成章节文本（先求完整再求质量） |
| Workshop 3 | 质量自检 | 检查 AI 痕迹、模板化表达、逻辑漏洞 |
| Workshop 4 | 润色打磨 | 语言精细化处理、排版优化、风格统一 |

Workshop 提示词与写作模式（modes/*.json）配合使用，支持 11 种题材模式：

| 模式名 | 适用题材 |
|--------|----------|
| general | 通用写作流程 |
| rule_mystery | 规则怪谈/悬疑推理 |
| urban_power | 都市异能/都市脑洞 |
| romance | 言情/古言 |
| folk_horror | 民俗灵异 |
| healing_life / healing_life_v2 | 治愈日常 |
| crazy_lit | 荒诞/癫狂叙事 |
| history_scholar | 历史/考据向 |
| reality_revenge | 现实复仇 |
| retro_life | 年代文 |
| female_solo | 女频单人叙事 |

## 五、核心技术特征

### 5.1 多引擎并联诊断

系统的每个章节诊断会同时经过：
1. **TextDiagnosticEngine** — 基于统计的 11 维特征向量评分
2. **BayesianAnalyzer** — 基于历史数据的质量后验概率
3. **CoxPrescriptionEngine** — 将问题转化为具体处方
4. **ReferenceEngine** — 从写作文脉库中检索可参考的同类问题处理方法

四份报告合并后，输出一份 **整合质量报告**，这与传统写作助手"只给分数不给答案"的模式有本质区别——系统不仅告诉你"哪里不好"，还告诉你"应该参考什么来改进"。

### 5.2 多Pass重写流水线

系统支持对章节进行 5 轮递进式重写：

```
P1 钩子密度 → P2 对话节奏 → P3 情绪平衡 → P4 动作密度 → P5 润色整合
```

每一步只聚焦一个核心问题，避免"一次改完所有问题反而越改越乱"。每轮输出独立保存，作者可回滚任意步骤（参见《末世：我有一座外星空间站》的 53 章正文目录结构，每章有 5×3=15 个历史版本）。

### 5.3 状态持久化

每个项目目录下均有 state.json 文件，记录：
- 项目元信息（标题、题材、字数、创建时间）
- 章节进度（已写/待写/已质检）
- 最近一次诊断结果缓存
- Workshop 阶段标记

这意味着系统可以随时从上次中断处恢复，而非每次从零开始。

### 5.4 参考资源的"零数据库"设计

ReferenceEngine 的核心设计决策是**避免数据库依赖**——所有资源通过文件系统直接加载：

- CSV 文件直接解析到内存 dict
- Markdown 文件直接读入
- 关键词检索使用 Python 内置 in 操作 + 关键词相似度匹配

这种设计让整个参考资源引擎可以被复制到任何 Python 3.8+ 环境立即运行，无需安装数据库或初始化脚本。

### 5.5 系统提示词分层

system_prompts/ 目录维护了 6 个独立的系统提示词文件，分别对应写作流水线的不同角色：

| 提示词 | 角色定位 |
|--------|----------|
| novel_writer | 主力作家——负责章节约束内的文本生成 |
| universal_writer | 通用写作——面向不挑题材的快速生成 |
| hook_expert | 钩子专家——章末悬念与情节反转 |
| quality_inspector | 质量督察员——Workshop 3 自检 |
| title_expert | 标题专家——标题/章节名优化 |
| market_analyst | 市场分析师——题材热度与爽点适配分析 |
| platform_rules | 平台规则——投稿合规性检查 |

### 6.2 已知问题与改进方向

基于项目诊断数据，系统存在以下可改进方向：

| 问题 | 影响范围 | 建议改进 |
|------|----------|----------|
| **情绪描写普遍缺失** | 17 个项目中普遍存在 | ReferenceEngine 已集成 emotion-psychology 指南，需要在 Workshop 2 中强制接入——每段场景后自动插入情绪检查 |
| **章末钩子持续缺席** | 全部项目的章节均未检测到 | 在 Workshop 2 草稿完成后，增加"章末钩子检查"后置步骤，未通过则不进入 Workshop 3 |
| **句长波动过大** | 规则怪谈类项目尤为明显 | 在润色阶段引入动态句长调整器（参考 style-adapter 的节奏规范） |
| **AI 模板词残留** | 约 50% 章节有 1–3 处残留 | Workshop 3 质量自检需接入 anti-ai-guide 的完整模板词库进行扫描 |
| **参考资源尚未写入时自动调用** | ReferenceEngine 只在手动查询时激活 | 需要在 novel_writer 的系统提示词中预埋资源检索指令——写作前先检索"相关技法"作为上下文注入 |
| **项目状态更新不实时** | state.json 只在 analyze 命令时刷新 | 引入 WorkflowOrchestrator（已有骨架）统一管理 |

### 6.3 技术债务

| 项目 | 说明 |
|------|------|
| knowledge/ 目录中存在多个 import_*.py 与 test_*.py 脚本（约 20+ 个） | 历史数据导入与测试遗留，可清理为 scripts/ 子目录统一管理 |
| 多个版本的主入口并存（pangu.py / pangu_plus.py / pangu_optimized.py / pangu_pipeline.py） | 建议以 pangu.py 为唯一主入口，其余标记为 deprecated |
| Java 后端无持续集成测试 | backend-java/ 的 Maven 项目缺少 JUnit 覆盖 |

## 七、总结与下一步建议

### 7.1 系统定位

盘古AI v3.0 是一个**"写作文脉工程化平台"**。它不同于市场上常见的 AI 写作产品（单轮生成一段文本），而是把整个创作过程视为一个可观测、可度量、可回放的工程流水线：

- **可观测** — 每个章节有 11 维特征评分 + 贝叶斯质量概率
- **可度量** — 有 739 条结构化写作技法作为参考标准
- **可回放** — 多Pass重写保留每轮版本，可随时回滚

### 7.2 资源资产总览

```
项目数: 17 个（共 100+ 章节）
参考文件: 101 个（CSV 9 + Markdown 92）
CSV技法条目: 739 条
写作模式: 11 种
题材模板: 37 种
Workshop阶段: 5 阶段
系统提示词: 6 套
```

### 7.3 下一步建议（按优先级）

2. **高优先级** — 增加章末钩子强制检查：Workshop 2 草稿通过后、进入 Workshop 3 之前，必须通过"钩子检测"

4. **中优先级** — 清理技术债务：将 knowledge/ 目录中 20+ 个历史脚本移入 scripts/ 子目录
5. **低优先级** — Java 后端增加 JUnit 测试覆盖，补充 CI/CD 配置
