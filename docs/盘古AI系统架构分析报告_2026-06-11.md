## 执行摘要

盘古AI是一款**多阶段流水线式AI小说写作引擎**，其核心理念是将"AI创作小说"这一复杂任务拆解为类似编译器的多道工序。系统创新性地设计了W0-W5六阶段写作管线、17层Prompt注入链、三层记忆系统、五路投影反馈机制以及三层关卡质控体系。

### 核心结论

> **架构设计已达专业级水准** —— W0-W5流水线、17层注入链、三层记忆、五路投影、三层关卡等设计理念在AI写作领域处于领先地位，充分借鉴了纹影(Rin'gateway)、OpenAI Memory Sandbox、LangChain等先进系统的设计思想。

> **核心问题在于实现层** —— 61KB单体文件、双管线并存、废弃代码堆积、RAG引擎缺失等问题严重影响了系统的可维护性和扩展性。

## 二、架构优势深度分析

### 2.1 W0-W5六阶段流水线设计

**核心设计思想**：将"创作一章小说"拆解为6道专业化工序，每道工序有明确的输入输出规范，支持灵活组合。

#### 流水线架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                     W0-W5 写作流水线                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    │
│  │  W0  │───▶│  W1  │───▶│  W2  │───▶│  W3  │───▶│  W4  │───▶│
│  │ 锚定 │    │ 准备 │    │ 初稿 │    │ 质检 │    │ 润色 │    │
│  └──────┘    └──────┘    └──────┘    └──────┘    └──────┘    │
│     │             │             │             │             │   │
│     ▼             ▼             ▼             ▼             ▼   │
│  主旨锚定      设定预处理      AI生成        逻辑质检      文笔精修 │
│  anchor       hotbank       draft         QC          polish   │
│                                                                │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    W5 提交阶段                          │    │
│  │  写入文件 → 更新state → 投影 → DB同步 → 记忆提交       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                │
└─────────────────────────────────────────────────────────────────┘
```

#### 双模式运行机制

| 模式 | 启用阶段 | 适用场景 | 特点 |
|------|----------|----------|------|
| **快速模式** | W0 + W2 + W4 | 日常连载、快速创作 | 效率优先，跳过复杂质检 |
| **工坊模式** | W0-W5 全部 | 关键章节、质量要求高 | 质量优先，全流程质检 |

**模式切换实现**：通过 `PipelineConfig.from_quick_mode()` 和 `from_workshop_mode()` 工厂方法实现配置的一键切换。

#### 闭环优化机制

W4润色阶段创新性地复用W3的QC反馈作为prompt的一部分，形成"生成→质检→润色→再质检"的完整闭环，确保质量持续提升。

### 2.2 17层Prompt注入链设计

**核心设计思想**：采用分层注入架构，将prompt构建拆解为17个独立层，每层有明确的职责边界，支持按需组合和独立调试。

#### 注入层架构

| 层级 | 名称 | 职责描述 | 启用条件 |
|------|------|----------|----------|
| L01 | 系统角色 | 定义AI在小说创作中的角色定位 | 始终启用 |
| L02 | 模式规则 | general/qimao/ershou/urban/fanfiction | 按项目配置 |
| L03 | 平台规则 | 七猫/二次元/知乎等平台特定规则 | 按平台配置 |
| L04 | 风格指引 | 调用style_fingerprint分析参考材料 | 可选 |
| L05 | 句式参数 | 平均句长/对话率/简体繁体/违禁词表 | 始终启用 |
| L06 | 参考材料 | 相关章节摘录（RAG检索结果） | 可选 |
| L07 | 前文摘要 | 前3章情节摘要（LLM生成） | 始终启用 |
| L08 | 角色状态 | 当前活跃角色的姓名/性格/状态/位置 | 始终启用 |
| L09 | 伏笔线索 | 开放伏笔列表（含状态和优先级） | 始终启用 |
| L10 | 故事合约 | MasterSetting + ChapterBrief 合同链 | 始终启用 |
| L11 | 记忆层 | Working(45%)/Episodic(30%)/Semantic(25%) | 始终启用 |
| L12 | RAG检索 | 混合型检索结果（向量+BM25融合） | 可选 |
| L13 | 节拍表 | 章节内节拍约束（5种模板） | 始终启用 |
| L14 | 章节任务 | 本章必须完成的具体任务 | 始终启用 |
| L15 | DB上下文 | 从SQLite读取的结构化上下文 | 可选 |
| L16 | 格式规则 | 输出格式要求（纯文本/分段/禁符号） | 始终启用 |
| L17 | 最终包装 | 最终prompt组装和清洗规则 | 始终启用 |

**架构价值**：
- **可组合性**：每层可独立开关，支持不同场景的prompt配置
- **可调试性**：可单独测试某一层的输出效果
- **可扩展性**：新增注入层只需实现统一接口
- **对齐先进系统**：与纹影(Rin'gateway)的Layer.System/Context/Task/Style/Output设计理念高度对齐

### 2.3 State↔DB双向同步设计

**核心设计思想**：采用"单一真相源(SSOT) + 索引镜像"架构，`state.json`作为权威数据源，SQLite作为索引化镜像支持高效查询。

#### 数据库表设计

| 表名 | 用途 | 关键字段 | 索引策略 |
|------|------|---------|----------|
| `character_states` | 角色状态追踪 | name, role, current_state, location, last_chapter | name, location |
| `foreshadowing_threads` | 伏笔线程管理 | thread_id, description, status, planted_ch, resolved_ch | status, planted_ch |
| `setting_constraints` | 世界观约束 | rule, category, status, source_chapter | category, status |
| `knowledge_entries` | Lorebook知识条目 | name, category, content, triggers, priority | category, priority |
| `ref_chapters` | 参考章节索引 | chapter_num, title, content, summary | chapter_num |

#### T02迁移机制

`db.py` 包含完整的Schema迁移代码，支持：
1. 自动检测旧版数据库结构
2. 重命名旧表 → 创建新表 → 迁移数据 → 删除旧表
3. 处理 `project_id`/`entry_type`/`book_id` 等历史遗留字段

#### 智能路由策略

`MemoryOrchestrator._route_query()` 根据查询类型自动分配预算：

| 查询类型 | Working | Episodic | Semantic |
|----------|---------|----------|----------|
| **write**（创作） | 50% | 30% | 20% |
| **review**（审核） | 20% | 50% | 30% |
| **query**（查询） | 10% | 30% | 60% |

#### 持久化策略

- **Working Memory** → `memory_scratchpad.json`（临时存储）
- **Episodic/Semantic** → SQLite `memory_entries` 表（持久存储）

## 三、关键问题深度诊断

### 3.1 双管线共存问题（风险：🔴 高）

**现状分析**：

项目中存在两套并行的管线引擎，形成显著的技术债务：

| 版本 | 文件 | 代码量 | 状态 |
|------|------|--------|------|
| **旧版** | `workflow_engine.py` | ~64.8KB | 已废弃但未清理 |
| **新版** | `pangu_core/pipeline.py` | ~21KB | 主力实现 |

**冲突点分析**：

```
旧版 workflow_engine.py:
  WorkflowEngine.run()
    → 调用自己的 Stage 基类.execute()
    → 使用 KnowledgeInjector 构建prompt（7层注入）
    → 使用 QualityGate 检查内容质量

新版 pangu_core/pipeline.py:
  WritingPipeline.run()
    → 调用 stages.py 的 W0-W5.execute()
    → 使用 prompt_builder.py 构建prompt（17层注入）
    → 使用 write_gates.py 执行关卡检查
```

**具体影响**：
1. **用户困惑**：新用户无法判断应使用哪个入口
2. **维护成本翻倍**：Bug修复需同时修改两套代码
3. **功能不一致**：KnowledgeInjector（7层）与PromptBuilder（17层）的prompt质量存在显著差异
4. **测试覆盖分裂**：两套管线需要分别进行测试

**修复建议**：

| 阶段 | 行动 | 预计工作量 |
|------|------|-----------|
| **Phase 1（立即）** | 在 `workflow_engine.py` 文件头添加 DEPRECATED 标记和迁移指引 | 0.5小时 |
| **Phase 2（本周）** | 检查旧版中是否有新版未覆盖的功能，如有则迁移 | 2小时 |
| **Phase 3（清理）** | 删除或移入 archive/ 目录保存 | 0.5小时 |

### 3.2 PromptBuilder单体膨胀问题（风险：🔴 高）

**现状分析**：

`prompt_builder.py` 单文件达到 **61KB，超过1500行**，包含全部17层注入逻辑，形成典型的"单体文件反模式"。

**问题表现**：
1. **定位困难**：修改某一层逻辑需要在巨大文件中搜索定位
2. **依赖隐式**：17层之间存在隐式依赖（如L13节拍表依赖L10合同链），但无显式声明
3. **测试困难**：无法单独测试某一层的注入逻辑
4. **合并冲突**：多人协作时易引发Git合并冲突

**重构方案：注册式注入器架构**

**目标结构**：
```
pangu_core/
  prompt_builder.py          ← 薄调度层（~200行）
  injectors/
    __init__.py             ← 注入器注册表
    base.py                 ← BaseInjector 抽象类
    system_role.py          ← L01-L03 系统角色与规则
    style_guide.py          ← L04-L05 风格与句式
    context_loader.py       ← L06-L08 上下文加载
```

**BaseInjector 接口设计**：
```python
class BaseInjector(ABC):
    @abstractmethod
    def get_layer_id(self) -> str:
        """返回层ID（如 "L01"）"""
        ...
    
    @abstractmethod
    def get_dependencies(self) -> List[str]:
        """返回依赖的层ID列表"""
        ...
    
    @abstractmethod
    def inject(self, context: PipelineContext) -> str:
        """执行注入，返回该层的prompt片段"""
        ...
    
    def is_enabled(self, config: PipelineConfig) -> bool:
        """该层是否启用（可被config禁用）"""
        return True
```

**PromptBuilder 调度逻辑**：
```python
class PromptBuilder:
    def __init__(self, config: PipelineConfig):
        self.config = config
        self.injectors = self._load_injectors()
    
    def _load_injectors(self) -> List[BaseInjector]:
        """从注册表加载启用的注入器，按依赖拓扑排序"""
        ...
    
    def build_prompt(self, context: PipelineContext) -> str:
        """按顺序调用每个注入器，拼接prompt"""
        prompt_parts = []
        for injector in self.injectors:
            if injector.is_enabled(self.config):
                part = injector.inject(context)
                prompt_parts.append(part)
        return "\n\n".join(prompt_parts)
```

**修复方案**：

**方案A（推荐）：实现轻量级FAISS向量检索**

```python
# pangu_core/rag_engine.py
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer
from pathlib import Path

class PanguRAG:
    def __init__(self, knowledge_dir: Path):
        self.knowledge_dir = knowledge_dir
        self.model = SentenceTransformer('paraphrase-multilingual-MiniLM-L12-v2')
        self.index = None
        self.documents = []
    
    def initialize(self):
        """构建FAISS索引"""
        self.documents = self._load_documents()
        embeddings = self.model.encode([d['text'] for d in self.documents])
        dimension = embeddings.shape[1]
        self.index = faiss.IndexHNSWFlat(dimension, 32)
        self.index.add(embeddings.astype(np.float32))
    
    def search(self, query: str, top_k: int = 10):
        """向量检索"""
        query_embedding = self.model.encode([query])
        distances, indices = self.index.search(query_embedding.astype(np.float32), top_k)
        return [self.documents[i] for i in indices[0]]
```
